# Skill Workspaces and Activation Gate

## Summary

Two related design pieces inside `SkillService` that took several rounds to reach their current shape: per-conversation skill workspaces (so files generated by tools in one chat don't leak into others), and an activation gate that prevents the AI from auto-discovering or reaching for skills the user hasn't explicitly enabled for the current conversation. Together they fix a class of footguns where Gilbert would silently use skills the user didn't expect, or worse, serve the wrong file from a stale workspace when the user clicked a download chip.

## Per-Conversation Workspaces

### Path layout

```
.gilbert/skill-workspaces/
    users/
        <user_id>/
            conversations/
                <conv_id>/
                    <skill_name>/         ← default for in-chat tool runs
                        generate_po.py
                        po.pdf
                        ...
    <user_id>/                            ← legacy single-workspace shape
        <skill_name>/                     (read-only fallback)
            ...
```

- New default: every `(user, skill, conversation)` triple gets its own directory under `users/<user_id>/conversations/<conv_id>/<skill_name>/`. Two parallel chats can call `write_skill_workspace_file("pdf", "po.pdf", ...)` simultaneously without colliding.
- Legacy: pre-refactor files under `<user_id>/<skill_name>/` are read-only fallbacks. New writes never go here. The workspace download handler tries the conv-scoped path first and falls back to legacy when an old attachment doesn't carry a `workspace_conv`.

### Resolver helpers

- `_workspace_root()` → `.gilbert/skill-workspaces`
- `_conversation_workspace_root(user_id, conv_id)` → `users/<user>/conversations/<conv>/`
- `_legacy_workspace_dir(user_id, skill_name)` → `<user>/<skill>/` (no create)
- `_get_workspace(user_id, skill_name, conversation_id=None)` — returns the conv-scoped path when conv_id is set, the legacy path otherwise. Always creates on demand.
- `_resolve_workspace_file(user_id, skill_name, rel_path, conversation_id)` — read-side helper that tries conv-scoped first then legacy, with a path-traversal check that fires against every candidate root regardless of whether it exists. Returns `(target_path, error_message)`.
- `_conv_id_from_args(arguments)` — pulls the injected `_conversation_id` off tool arguments, returns `None` for system callers.

### Tool wiring

Tools read `_conversation_id` from their injected arguments and thread it through to the resolver helpers:

| Tool | Workspace path used |
|---|---|
| `read_skill_file` | (skill bundled files — uses skill_dir, not workspace) |
| `run_skill_script` | conv-scoped as cwd, falls back to legacy via `_get_workspace` |
| `browse_skill_workspace` | conv-scoped primarily; falls back to legacy when conv-scoped is empty |
| `read_skill_workspace_file` | uses `_resolve_workspace_file` (conv-then-legacy) |
| `write_skill_workspace_file` | conv-scoped only — never writes to legacy |
| `run_workspace_script` | conv-scoped only |
| `attach_workspace_file` | conv-scoped only; stamps the resulting `FileAttachment` with `workspace_conv` so the download handler can find it |

### `FileAttachment.workspace_conv`

Added to `interfaces/attachments.py`. Reference-mode attachments now carry three fields: `workspace_skill`, `workspace_path`, `workspace_conv`. Empty `workspace_conv` means legacy (pre-refactor) — the download handler falls back to the per-(user, skill) path. Round-trips through `_serialize_message` / `_deserialize_message` and `_serialize_attachments_for_wire`.

### WS download handler

`skills.workspace.download` accepts an optional `conversation_id` frame field. The frontend's `downloadSkillWorkspaceFile(skillName, path, conversationId?)` reads `attachment.workspace_conv` and passes it. The handler tries the conv-scoped workspace first and falls back to legacy. Path-traversal check fires for both candidates.

### Shared-room cross-member fallback

Workspaces are per-user-per-conv on disk (`users/<uid>/conversations/<cid>/`), so an upload Dylan dropped into a shared room with Root lives under Dylan's path — Root opening the same attachment can't find it under his own workspace root and the lookup 404s.

`WorkspaceService.member_workspace_roots(caller_user_id, conv_id) -> list[Path]` widens the search by returning the workspace roots of *other* members for shared conversations (empty list otherwise). Both call sites use it:

- `_ws_workspace_download` (general workspace files via the `skills.workspace.download` WS RPC) appends member roots to its candidate list after the per-user and legacy entries.
- The HTTP route `GET /api/chat/download/{conv_id}/{path}` (chat-uploads only, served by `gilbert/web/routes/chat_uploads.py`) appends `member_root/uploads/<safe_name>` to its candidates and adds each member root to the path-escape allowlist.

The widening is gated on `check_conversation_access(conv_data, caller_ctx)` returning `None` — the same gate the chat-read paths use — so a caller who can't see the conv can't reach its members' workspaces via this code path either. Personal conversations short-circuit to an empty list (no DB read), keeping the common case free.

The protocol method lives on `WorkspaceProvider` (`gilbert.interfaces.workspace`), so route + service share one implementation instead of duplicating the conv-load + member-walk in each layer.

### Cleanup hook

When `_ws_conversation_delete` removes a personal conversation, `AIService` publishes `chat.conversation.destroyed` with `{conversation_id, owner_id}`. `SkillService.start()` subscribes to that event and calls `_on_conversation_destroyed`, which:

1. Resolves the conv-scoped workspace root for the named owner.
2. For shared rooms (no `owner_id`), walks all per-user `users/<u>/conversations/<conv>/` candidates.
3. **Defense in depth:** refuses to `rm` anything whose resolved path doesn't sit inside `_workspace_root().resolve()`. A crafted conv_id can't escape the workspace tree.
4. `shutil.rmtree(target, ignore_errors=True)` and logs the deletion.

`SkillService.stop()` unsubscribes the handler. Tests use `_on_conversation_destroyed` directly to verify cleanup without needing the bus plumbing.

## Activation Gate

### Two-layer semantics

Skill activation in this codebase has dual semantics:

1. **Soft signal (always was):** activated skills get their `SKILL.md` instructions injected into the system prompt via `SkillService.build_skills_context()`. Activation is per-conversation, stored on the conversation row's `state.active_skills`.
2. **Hard gate (new):** AI-driven calls to skill tools refuse when the skill isn't on the conversation's active list. The refusal is a JSON error string telling the AI to ask the user to enable the skill instead of retrying. Slash invocations bypass the gate entirely (user-initiated → explicit "use this skill" signal). System callers (no conversation_id) also bypass.

### How the gate is wired

- Both `_execute_tool_calls` and `_execute_slash_command` inject two new underscore-prefixed args alongside `_user_id` etc.: `_conversation_id` and `_invocation_source` (`"ai"` or `"slash"`).
- `SkillService._assert_skill_accessible(skill_name, arguments) -> str | None` is the gate helper. Returns `None` (allow) when invocation is slash, when no conv_id is set (system callers), or when the skill is on the conv's active list. Otherwise returns a JSON error string the tool returns verbatim, instructing Gilbert to ask the user to enable the skill and not retry the same skill_name in this turn.
- Called at the top of every gated tool method: `read_skill_file`, `run_skill_script`, `browse_skill_workspace`, `read_skill_workspace_file`, `write_skill_workspace_file`, `run_workspace_script`, `attach_workspace_file`. Not gated: `manage_skills`, `create_skill`.
- `manage_skills(action=list)` is filtered for the AI separately: when invoked via the AI path on a conversation context, returns only skills active for that conversation. Slash invocations and system callers see the full catalog.
- The system prompt's "## Skills" section explicitly tells Gilbert that skill tools are gated, NOT to call them speculatively to discover skills, and to ask the user to enable a skill from the panel if it needs one. The gate is the hard backstop; the prompt is the soft hint that prevents the AI from even trying.

### Slash-command escape hatch

The user typing `/skill wsread pdf foo.txt` always works regardless of activation — slash commands are explicit user intent, not AI-driven discovery. The gate's `_invocation_source != "ai"` short-circuit handles this. This preserves the manual escape hatch while preventing Gilbert from auto-using skills the user didn't ask for.

### Why this exists

Brian noticed: a fresh chat where he hadn't enabled the pdf skill was using it anyway. Gilbert was discovering it via the system prompt's "## Skills" section + `manage_skills(list)` and reaching for the tools directly. The gate forces deliberate user opt-in for AI use while leaving everything available for direct slash invocation.
